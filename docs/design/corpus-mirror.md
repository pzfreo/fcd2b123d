# Design: a mirrored corpus of FreeCAD library parts with reasoning traces

**Status:** forward-looking design — captured from a prior design discussion for future implementation. Not yet acted on.

## Goal

Translate a large subset of the FreeCAD Parts Library (3,194 files; ~70% type-level addressable as of #41) into a self-contained, publishable dataset that pairs each source `.FCStd` with:

1. **Executable build123d Python** (the translator's output today).
2. **Structured feature trace** — deterministic record of how the FreeCAD designer built the part, derived from the existing `ctx.add_step(...)` chain.
3. **Structured reasoning trace** — LLM-augmented chain-of-thought capturing *why* each step exists.
4. **Natural-language narrative** — human-readable explanation.
5. **Provenance metadata** — source hash, license attribution, exporter version, regeneration timestamps.

Two consumer audiences anchor the design:

- **Researchers / LLM trainers** wanting `(natural-language goal → build123d code)` pairs as fine-tuning data.
- **build123d users** wanting working code examples for fasteners, brackets, profiles, etc.
- **CAD reasoning research** — few well-structured datasets pair parametric history with explanations.

## Core idea: mirror the upstream hierarchy

The dataset directory structure mirrors the FreeCAD Parts Library's structure 1:1. Each source file gets a parallel set of derived artifacts in the same path.

### Why this works

- **Self-documenting.** Anyone reading `Mechanical Parts/Fasteners/Bolts/M6x20.FCStd` in the source finds `Mechanical Parts/Fasteners/Bolts/M6x20.py` here. No index needed to navigate.
- **Multiple artifacts serve multiple consumers.** Same file → researchers want the JSON, build123d users want the `.py`, LLM trainers want the prose CoT. One repo serves all three.
- **Diff-friendly.** When upstream updates a part, regeneration shows exactly what changed in derived artifacts.
- **License is clean.** FreeCAD-library parts are CC-BY 3.0 ([source](https://github.com/FreeCAD/FreeCAD-library)). Derivative datasets are compatible as long as authorship is preserved — record commit authorship in each artifact's metadata.

### Per-model file layout

```
Mechanical Parts/Fasteners/Bolts & Screws/
├── ISO4017-M6x20.FCStd              ← upstream source (vendored, Git LFS)
├── ISO4017-M6x20.py                  ← build123d module, executable
├── ISO4017-M6x20.features.json       ← deterministic feature extraction
├── ISO4017-M6x20.cot.json            ← structured reasoning trace
├── ISO4017-M6x20.cot.md              ← natural-language narrative
└── ISO4017-M6x20.meta.json           ← provenance, hashes, exporter version
```

### Why split features / cot / meta into separate files

Splitting feels verbose but pays off:

- **`features.json`** is regenerated by the deterministic translator (this codebase). Updates whenever the translator changes.
- **`cot.json` / `cot.md`** are regenerated by an LLM pass over `features.json` + the source. Update on different cadence and review process.
- **`meta.json`** tracks provenance (source hash, exporter version, dates). Updates on every regeneration.

If everything were one JSON, every LLM-prose regeneration would create noisy diffs in the deterministic data. Three files = three review tracks.

## Decisions to make before reaching ~2000 files

These compound — making them later means rewriting a lot.

### 1. Vendor or submodule the upstream `.FCStd` files?

| Option | Pros | Cons |
|---|---|---|
| **Vendor** (copy into this repo, use Git LFS for blobs) | Self-contained, reproducible, citable — the dataset stands alone | Bloats repo, sync burden when upstream changes |
| **Submodule** (point at upstream commit) | Keeps repo small, clean upstream tracking | Breaks if upstream rewrites history or goes offline; not citable as-is |

**Recommendation: vendor with Git LFS.** Publishable datasets must be self-contained.

### 2. Exporter version pinning

Every generated artifact records which translator version produced it:

```json
// in meta.json
{
  "exporter_version": "0.3.1",
  ...
}
```

When the translator improves, this lets a regenerate script find every file needing rebuild via `grep` or a query. Without this, you have no way to ask "which files were generated by the v0.2 translator that had the snap bug?"

### 3. Regeneration as a first-class operation

Build a `tools/regenerate.py` early:

- `regenerate.py path/to/M6x20.FCStd` — rebuilds all derived artifacts for one source.
- `regenerate.py --all` — entire corpus.
- `regenerate.py --changed-since v0.2.0` — diff against a version.
- `regenerate.py --missing-cot` — files with no `.cot.json`.

You'll run this constantly. Worth the early investment.

### 4. CI validates every commit

For each model, CI must check:

- The `.py` executes without error.
- The geometry hash matches `meta.json`'s recorded hash.
- Each JSON conforms to its schema (JSON Schema files in `/schemas`).
- The `.cot.md` parses as well-formed Markdown with the expected front-matter.

Catching regressions across 2000 files manually is impossible; CI is non-negotiable.

### 5. Schema versioning

```json
// in every JSON artifact
{
  "schema_version": "1.0",
  ...
}
```

Schema changes are inevitable. Versioning keeps old records interpretable. JSON Schema files in a top-level `/schemas/` directory, validated in CI.

## Top-level index for queries

Walking 2000+ nested files to filter is slow. Generate a single `index.jsonl` at the corpus root with one line per model:

```jsonl
{"path": "Mechanical Parts/Fasteners/Bolts/M6x20", "category": ["Mechanical Parts", "Fasteners", "Bolts"], "features": ["revolve", "pad", "pocket", "chamfer"], "param_count": 4, "source_hash": "sha256:abc...", "exporter_version": "0.3.1"}
{"path": "Architectural Parts/Hydro equipment/Faucets/Lorenzetti", "category": [...], ...}
```

This is what tooling and any future MCP server load. Per-model files stay readable; the index makes the dataset queryable in one `jq` invocation.

## Natural-language CoT file format

Structured Markdown rather than free prose:

```markdown
# ISO 4017 M6×20 Hex Bolt

**Category:** Mechanical Parts / Fasteners / Bolts & Screws
**Features:** 4 (revolve, pad, pocket, chamfer)
**Parameters:** thread_diameter, length, head_height, head_across_flats

## Construction reasoning

### Step 1 — Establish the shaft profile
The shaft is a body of revolution, so we begin with a 2D profile on the XZ plane...

### Step 2 — Add the hex head
...
```

Machine-parseable structure (headers, metadata block) while reading as a tutorial. Same content as `cot.json` but optimised for human consumption — different downstream uses want different forms.

## Trap: don't let prose drift from structured data

The natural-language `.cot.md` and the structured `.cot.json` must be generated from the same source-of-truth trace. If a human edits the `.md` for clarity, that change should:

- Either propagate back to the JSON (round-trip the structured trace).
- Or be explicitly marked as a prose-only refinement (so future regeneration knows what to preserve vs overwrite).

Otherwise you accumulate subtle inconsistencies that poison the dataset for training and analysis.

## Why this is novel

Few well-structured CAD reasoning datasets exist. The combination of:

- Mirrored hierarchy
- Executable code
- Structured features
- Structured reasoning
- Natural-language reasoning
- Provenance
- CI-validated

is genuinely new. Even at a few hundred models it's publishable. At 2000+ it's a real resource.

## README is part of the product

Datasets get adopted based on whether the README convinces a reader in 60 seconds. The corpus README must:

- Explain *why* each derived artifact exists.
- Identify the intended consumers.
- Show one fully-worked example on the front page. The **M6 bolt is the ideal hero** — small enough to display in full, recognisable enough that readers can verify the reasoning by eye.

Write the README with the same care as the code.

## Open questions worth deeper design before implementation

1. **CoT JSON schema** — what exactly is the structured shape of "reasoning"? A list of `{step, intent, geometric_consequence}` records? Pointers into the features.json indices? Has its own conversation worth.
2. **CI/regeneration pipeline** — how to keep `cot.*` regenerable cheaply (LLM cost), and what's the cache/invalidation model? Also worth its own conversation.
3. **Subset-vs-whole-library strategy** — vendor all 3,194 files or only the ones we can currently translate accurately (~62%)? The honest dataset includes failures with `status: unsupported` markers; the curated dataset only ships passing examples. Both have value; pick deliberately.
4. **License handling** — how to surface CC-BY-3.0 attribution per-file (each `meta.json` records original author and source URL) vs in aggregate (top-level `ATTRIBUTION.md`).
5. **Relationship to `tests/fixtures/sample_813/`** — the recent 100-file audit already created a small-scale version of this structure (manifest + report + per-fixture artifacts). Worth keeping that structure in mind when shaping the larger corpus.
