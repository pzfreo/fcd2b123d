# Generated-code quality: pros, cons, action plan

A review of what `fcstd2b123d` currently emits, measured against idiomatic
build123d as written in [`bd_warehouse`](https://github.com/gumyr/bd_warehouse)
(the curated component library by build123d's author). Reviewer hat:
senior Python + 10-yr-CAD engineer; assume the consumer is either a human
who wants to tweak dimensions or an LLM that wants to refactor.

The bar is **"could a competent build123d developer mistake this for
hand-written code?"** — the answer today is *no*, but it's closer than
the FreeCAD-emit-Python tools that came before, and there's a clear path
to *yes* without changing the translator's architecture.

---

## What we emit today (representative samples)

`tier4_patterns/polar_pattern_holes.FCStd`:

```python
"""Auto-generated by fcstd2b123d from tests/fixtures/tier4_patterns/polar_pattern_holes.FCStd."""

from build123d import Circle, Pos, Rot, extrude


def _pattern_difference(base, *removals):
    """Boolean-subtract each removal from ``base`` via BuildPart..."""
    from build123d import BuildPart, Mode, add
    with BuildPart() as _bp:
        add(base)
        for s in removals:
            add(s, mode=Mode.SUBTRACT)
    return _bp.part


# Sketcher::SketchObject 'Disc': 1 circle (1 loops)
disc = Circle(25)

# PartDesign::Pad 'Pad': length=6.0
pad = extrude(disc, amount=6)

# Sketcher::SketchObject 'Hole': 1 circle (1 loops)
hole = Pos(18, 0) * Circle(2)

# PartDesign::Pocket 'Pocket': length=12.0 (reversed)
pocket = pad - extrude(hole, amount=12)

# PartDesign::PolarPattern: step=60.0deg, occurrences=6
polar_pattern = _pattern_difference(
    pocket,
    Rot(Z=60) * extrude(hole, amount=12),
    Rot(Z=120) * extrude(hole, amount=12),
    Rot(Z=180) * extrude(hole, amount=12),
    Rot(Z=240) * extrude(hole, amount=12),
    Rot(Z=300) * extrude(hole, amount=12),
)

result = polar_pattern
```

A bd_warehouse-style author would write the same model as roughly:

```python
"""6-hole disc — pierced by a polar-patterned ⌀4 mm hole."""
from build123d import BuildPart, BuildSketch, Circle, Pos, PolarLocations, extrude, Mode

with BuildPart() as disc:
    Circle(25)              # outer profile
    extrude(amount=6)
    with PolarLocations(18, 6):                 # 6 holes on R=18
        with BuildSketch() as hole:
            Circle(2)
        extrude(amount=-12, mode=Mode.SUBTRACT)

result = disc.part
```

Same geometry, ~⅓ the lines, and the `6 hole pattern` reads directly from
the structure rather than being spelled out tooth-by-tooth.

## Pros — what we already get right

| # | Item | Notes |
|---|---|---|
| 1 | **Specific named imports** | Never `from build123d import *`. Matches bd_warehouse's main-module style. |
| 2 | **Provenance comments per feature** | Every emitted unit carries a `# PartDesign::Pad 'Pad': length=6.0` comment that traces back to the FreeCAD source object. Invaluable for debugging. |
| 3 | **Stable `result =` final binding** | Test contract + REPL ergonomics in one. |
| 4 | **Module helpers emitted on demand** | `_edges_at`, `_pattern_union`, etc. only appear when used. No bloat in simple files. |
| 5 | **black-formatted output** | Consistent style, no whitespace bikeshedding. |
| 6 | **snake_case post-pass with shadow protection** | `chamfer = chamfer(...)` is rewritten to `chamfer_0 = chamfer(...)` so the imported function isn't masked. (PR #54, "draft" added in PR #70.) |
| 7 | **Parametric models become functions** | Tier-6 spreadsheet models render as `def make_part(width=25, depth=15, height=10): ...` with defaults — a clean parametric API a human can call. |
| 8 | **Algebra mode is read-aloud-able** | `pad - extrude(hole, amount=-12)` is mathematically transparent. For simple files this beats `with BuildPart() as p: ...`. |

## Cons — gaps vs. expert hand-written

### 1. Patterns are spelled-out instead of structured

Every polar / linear / mirror copy is emitted as a separate `Rot(...) *`
or `Pos(...) *` term inside a multi-arg helper call. bd_warehouse uses
`with PolarLocations(radius, count): …` / `with GridLocations(…): …` so
the *intent* is in the code, not just the geometry. **High impact** —
this is the single biggest "looks autogenerated" tell in our emit.

**Where it hits**: tier4_patterns (every fixture), every polar/linear/mirror
inside a Body, any sprocket-like part.

### 2. No semantic naming — variables echo FreeCAD object names

`sketch_001_profile`, `pocket_002`, `chamfer_0`. The original FreeCAD
object names are preserved (good for traceability), but they carry no
semantic information about what the variable *means*. Hand-written code
uses `hex_head_profile`, `shank`, `chamfer_tip`. **Medium impact** —
hurts readability and LLM-comprehension; doesn't affect correctness.

### 3. Solver-noise digits in coordinates

`Pos(54.999978668734606, 15) * Circle(25)` — the user typed `55` in
FreeCAD's GUI and the constraint solver settled at `54.99997866…`.
We pass the noise through verbatim. **Known issue (#43)** — naive snap
breaks BRep validity because the arc-end was matched to the line-start
at full precision; needs the anchor-point refactor outlined in #43.
**High impact on perception, zero impact on correctness.**

### 4. Sketch-on-plane wrap is verbose

```python
sketch_001 = Sketch() + Plane(origin=(0, 0, 5.2), x_dir=(-1, 0, 0), z_dir=(0, 0, 1)) * (
    Circle(7.41580486637) - make_face(sketch_001_profile)
)
```

bd_warehouse writes the same with builder-mode:

```python
with BuildSketch(Plane.XY.offset(5.2)) as sketch_001:
    Circle(7.41580486637)
    make_face(sketch_001_profile, mode=Mode.SUBTRACT)
```

**Medium impact** — line-count savings + cleaner intent. The `Sketch() +`
wrap exists for mypy reasons (see comment in `sketch.py`) but is line noise.

### 5. No `MM` / `IN` units

Every literal is a bare float. bd_warehouse multiplies by `MM` or `IN`
constants (which are just `1.0` and `25.4`) so the *unit story* is
visible. Mechanical CAD lives or dies by unit confusion. **Low impact in
practice** (FreeCAD is mm-native) but it's a missed legibility win.

### 6. Module-level imperative everywhere

bd_warehouse parts are **classes** subclassing `BasePartObject`, with
`__init__` taking size args and `super().__init__(part, label=..., color=…)`
attaching joints and metadata. Our emit is a flat sequence of
statements with no encapsulation. **Low-ish impact** for our use case
(translate-and-run) — but it means our output can't be `imported` and
re-used as a parametric component the way a bd_warehouse part can.

### 7. Helpers are duplicated across files

`_edges_at`, `_pattern_union`, `_pattern_difference`,
`_pattern_intersection`, `_faces_at` get inlined into every module that
uses them. A 30-line helper duplicated 50 times is 1500 lines of clutter
in a multi-file generated codebase. **Medium impact** when the user is
consuming multiple translations.

### 8. No docstring beyond "Auto-generated"

bd_warehouse parts open with a multi-line description, parameter doc,
and raise list. Ours says only "Auto-generated by fcstd2b123d from
…path.FCStd." — fine for traceability but useless if the consumer
deletes the source path. **Low impact** but easy to improve.

### 9. Tessellation-precision floats everywhere

`start_angle=270.00003491975656`, `arc_size=261.7867543785052`. Same
root cause as #3 — solver settlement — but appears even when the user
*didn't* type a round number, because FreeCAD's solver targets numerical
satisfaction not authorial intent. We have no signal to tell the cases
apart, so we can't safely round.

## Action plan — phased by ROI

### Phase 1 — Cosmetic wins, no architectural change (≈ 2-4 days)

| # | Change | Effort | Payoff |
|---|---|---|---|
| 1.1 | Detect `Rot(Z=k·θ)` chains in patterns and emit `with PolarLocations(r, n): …` instead | 1 day | Single biggest "doesn't look autogen" change |
| 1.2 | Same for `Pos(i·dx, 0, 0) * …` → `with GridLocations(dx, 0, n, 1): …` | ½ day | Tier-4 linear patterns |
| 1.3 | Add `MM = 1.0` / `IN = 25.4` constants when emitting and multiply literals by them (gated on a CLI flag for now to avoid breaking tests) | ½ day | Legibility |
| 1.4 | Module docstring auto-generated from FreeCAD `Doc.Label` + `Doc.Comment` (when set), falling back to current "Auto-generated" line | ½ day | Self-documenting output |
| 1.5 | Co-locate the per-feature comment with the *intent* not just the property dump (`# Hexagon recess for socket screw head`, sourced from the FreeCAD object's `Label`) | ½ day | Already half there — the `Label` is in our comment, just buried |

### Phase 2 — Semantic naming (≈ 1-2 days)

| # | Change | Effort | Payoff |
|---|---|---|---|
| 2.1 | When the FreeCAD `Label` differs from `Name` ("Hexagon recess" vs "Sketch001"), promote `Label` to the variable name (snake-cased) and demote `Name` to a comment | 1 day | Names go from `sketch_001` to `hexagon_recess` |
| 2.2 | Detect "the last solid" pattern and name it `result_<bodylabel>` instead of `pad_003` | ½ day | More skim-able files |

### Phase 3 — Helpers via shared module (≈ ½ day)

| # | Change | Effort | Payoff |
|---|---|---|---|
| 3.1 | Add a CLI option `--helpers-import-from <pkg>` that emits `from .fcstd2b123d_helpers import _edges_at, …` instead of inlining each helper | ½ day | Multi-file output is 10× cleaner |
| 3.2 | Ship a tiny `fcstd2b123d.runtime` module with the helpers, installable as a dependency of any consumer of generated code | already done — just expose it | One-import, no duplication |

### Phase 4 — Builder-mode for sketches (≈ 2-3 days, opt-in)

| # | Change | Effort | Payoff |
|---|---|---|---|
| 4.1 | Add a `--style=builder` emit mode that wraps each sketch in `with BuildSketch(plane) as s: …` and uses `Mode.SUBTRACT` rather than `-` | 2 days | bd_warehouse-style output; opt-in so the existing algebra-mode contract stays |
| 4.2 | Same for body chains: `with BuildPart() as part: …` with `add(...)` and `Mode.ADD/SUBTRACT` | 1 day | Final emit reads almost like hand-written |

### Phase 5 — Coherent snap (deep, defer per #43)

The "54.999978" issue. Real fix needs the anchor-point refactor
described in #43: shared vertices between adjacent edges must snap
*together* or stay un-snapped *together*. Out of scope for a
cosmetic pass; tracked separately.

### Phase 6 — Classes (deferred / philosophical)

bd_warehouse subclasses `BasePartObject`. We *could* emit a class per
top-level Body but it costs us the "translate-and-run" property — the
consumer would need to import build123d to even import the file. Punt
unless a clear use case appears.
