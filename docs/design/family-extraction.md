# Family extraction and class-based emission

**Status**: design / accepted, implementation pending
**Last updated**: 2026-05-17
**Audience**: project maintainers and future agents working on either
`fcstd2b123d` or `bd-freecad-library`

## What this document is

A consolidated architecture for **how parametric build123d classes get
produced from the FreeCAD Parts Library** — replacing the ad-hoc
hand-writing workflow that bootstrapped `bd-freecad-library` in v0.x
with a deterministic, manifest-driven pipeline coded into the
translator.

Supersedes (or absorbs) parts of:
- [`fcstd2b123d#118`](https://github.com/pzfreo/fcd2b123d/issues/118)
  (class-based parametric component emission) — this doc is the
  consolidated vision; #118 stays open as the implementation tracker.
- `docs/strategy/derivative-libraries.md` — the strategy doc described
  hand-writing parametric Python in `bd-freecad-library`. That's the
  bootstrap workflow (v0.x); the steady-state workflow described here
  is manifest-driven.

## The core principle

**LLM judgement is not a stable substrate for a repeatable, testable,
maintainable pipeline.**

If a workflow depends on "have an LLM look at these N fixtures and
write a parametric class", we never get:
- the same parameter names across iterations,
- the same parameter ordering,
- the same simplification choices,
- the same default values.

The fix is to move the deterministic logic into the translator (which
can be tested) and the semantic judgement into **declarative manifests
that are version-controlled and code-reviewed** (which can be diffed
and re-applied). The LLM has a documentation/scaffolding role; it is
never *in* the pipeline.

## The architecture

### 1. Class-based emission (`--style=class`)

The translator gets a new emit mode alongside the existing `auto` /
`algebra` / `builder`. For a fixture, `--style=class` emits:

```python
"""Auto-generated by fcstd2b123d from <fixture>."""

from build123d import Align, BasePartObject, Mode, Part, RotationLike, ...


class GeneratedPartName(BasePartObject):
    """<one-line summary derived from FreeCAD doc Label or filename>."""

    def __init__(
        self,
        rotation: RotationLike = (0, 0, 0),
        align: Align | tuple[Align, Align, Align] | None = None,
        mode: Mode = Mode.ADD,
    ):
        # ... translated build123d body ...
        super().__init__(part=<result>, rotation=rotation, align=align, mode=mode)
```

The body contains hard-coded dimensions (a single fixture → a single
specific-instance class). The output matches `bd_warehouse`'s class
shape exactly: standard rotation / align / mode kwargs, `BasePartObject`
inheritance, predictable docstring location.

For a body chain, the body's translated code (using `with BuildPart()`
or algebra, depending on suitability) sits inside `__init__`.

### 2. Family manifests

A manifest is a YAML file that declares a parametric family:

```yaml
family: en_10058_flat_bar
class_name: EN10058FlatBar
standard:
  ref: EN 10058:2018
  title: Hot rolled flat steel bars for general purposes — Dimensions and tolerances on shape and dimensions
  url: https://www.en-standard.eu/csn-en-10058-...

# Where the fixtures live (glob, relative to fcstd2b123d/tests/fixtures/).
fixture_glob: "**/Flat_Bar_*_EN10058_S235JR.FCStd"

# Regex to extract parameters from each fixture's filename.
# Named groups become parameters of the generated class.
filename_pattern: "Flat_Bar_(?P<width>\\d+)x(?P<thickness>\\d+)_EN10058_S235JR"

# Per-parameter declarations.
parameters:
  - name: width
    source: filename       # filename | extrude_amount | spreadsheet:<cell> | constant
    type: float
    units: mm
    doc: "Larger cross-section dimension."
  - name: thickness
    source: filename
    type: float
    units: mm
    doc: "Smaller cross-section dimension."
  - name: length
    source: extrude_amount # extracted from the extrude(...) constant
    default: 50
    type: float
    units: mm
    doc: "Bar length along longitudinal axis."

# Optional: declarative cross-references for documentation.
docstring: |
  EN 10058 hot-rolled flat steel bar — rectangular cross-section
  (width × thickness) extruded along the bar's longitudinal axis.

  See ``docs/standards/en_10058/`` for the dimension table and rationale.
```

Schema fields:

| Field | Required | Description |
|---|---|---|
| `family` | yes | Slug used in generated filenames |
| `class_name` | yes | The Python class name |
| `standard.ref` | yes | E.g. `EN 10058:2018` |
| `standard.title` | optional | Standard's full title |
| `standard.url` | optional | Public URL |
| `fixture_glob` | yes | Glob (relative to fixtures root) of source fixtures |
| `filename_pattern` | optional | Regex with named groups; required when params come from filename |
| `parameters` | yes | List of declarations; each has `name`, `source`, `type`, `default` (if applicable), `doc` |
| `docstring` | optional | Multi-line docstring for the generated class |
| `base_class` | optional | A `BasePartObject` subclass to inherit from (deferred to phase 6) |

### 3. Family extraction algorithm

Given a manifest + corpus, the algorithm:

1. Glob the fixtures matching `fixture_glob`.
2. For each fixture:
   - Parse parameters from its filename (or other sources) per the manifest.
   - Translate via `--style=class` (Phase 1 above) into specific-instance class code.
3. Diff the N translated outputs:
   - Walk all N ASTs in parallel.
   - At each numeric-literal position, collect the per-fixture values.
   - For each position, check whether the literal is expressible as a
     **simple function of the fixture's parameter values** — literal
     equality, half, double, negation, sum, etc. — using the per-fixture
     parameter dict from step 2.
   - If a substitution holds across **all** fixtures: replace the
     literal with the parameter reference (e.g. `60` → `width`,
     `-30` → `-width/2`).
   - If a literal is constant across all fixtures (e.g. all 22 fixtures
     have `amount=50`): keep as default value of a parameter declared
     with `source: extrude_amount`.
   - If a literal varies but doesn't fit any simple function of declared
     parameters: this is a manifest error (the family has hidden
     parameters the manifest didn't declare). Fail with a clear
     diagnostic.
4. Emit one parametric class with `__init__(width, thickness, length=50,
   rotation, align, mode)` whose body has parameter references in
   place of the relevant literals.
5. Validate: instantiate the generated class with each fixture's
   parameters and verify the geometry matches the per-fixture
   specific-instance class (the existing harness already does this).

### 4. `bd-freecad-library` workflow

```bash
# Author or edit the manifest:
$EDITOR families/en_10058.yaml

# Regenerate the Python (in repo's build step or pre-commit hook):
fcstd2b123d-family families/en_10058.yaml \
  > src/bd_freecad_library/standards/en_10058.py

# Verify:
pytest tests/test_en_10058.py
```

The generated `en_10058.py` **is checked in**, so library consumers
don't need `fcstd2b123d` at install time. The build step is part of
the contributor workflow, not the user workflow.

`tests/test_en_10058.py` itself is also generated from the manifest
(or trivially small — `validate_family(EN10058FlatBar, manifest)`).

## The LLM's role

| Step | LLM role |
|---|---|
| This design doc | **Drafted by LLM**, then human-reviewed and edited before commit |
| Phase 1-3 implementation code | None — same as any other PR (humans review) |
| Family manifest authoring | **LLM drafts** given a corpus query; **human reviews** before commit |
| Code generation | None — the translator does it deterministically |
| Validation | None — the harness does it |
| Documentation tweaks | LLM helpful, human reviews |

**Rule**: the LLM never sits *in* the deterministic pipeline. Same
input → same output, with or without the LLM in the loop. The LLM is
a once-per-task scaffolding tool whose artifacts get human-reviewed
before becoming part of the durable system.

## Irreducible boundaries

Even with everything above, three things stay outside what the
translator can decide:

1. **Family identification is semantic.** The translator can cluster
   fixtures by filename pattern and TypeId structure, but it cannot
   *decide* which clusters constitute a "family." That decision lives
   in `families/<name>.yaml` (one manifest = one declared family).
2. **Parameter naming is convention.** The translator can detect that
   `60` is a parameter (varies across fixtures, fits `width`); it
   cannot decide the canonical name. The manifest names it.
3. **Standards metadata is engineer-authored.** Standard reference,
   units, docstring, default rationale — these live in the manifest.

These aren't algorithmic failures; they're the natural boundary
between syntax and semantics. Encoding them in version-controlled
manifests, not LLM judgements, is the goal.

## Open questions (defer until implementing)

- **Complex parametric expressions.** What's the right syntax for a
  manifest to declare that "this literal is `width * cos(angle)`"?
  Defer until the simple linear substitution doesn't suffice on a real
  family.
- **CSV-driven dimension tables.** `bd_warehouse` loads CSVs at
  runtime. Should our generated classes do the same, or inline the
  table as a Python dict? Probably inline for now; CSV is a phase-6
  feature.
- **Validation of manifest correctness.** Beyond schema validation:
  should the family-extraction tool refuse to emit if the substitution
  fails for any fixture? Yes — fail loudly, force the manifest to be
  fixed.
- **What happens when fixtures span a TypeId mix?** Probably error;
  same-family fixtures should have isomorphic TypeId structure.
