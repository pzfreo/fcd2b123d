# Emit style guide

How `fcstd2b123d` should write build123d Python. Reference for anyone
adding a translator or reviewing emitted code.

The single test we apply is the **"hand-written?" check**: would a
build123d engineer reviewing one of our outputs in isolation, without
the `Auto-generated by …` header, mistake it for code a colleague wrote?
The closer the answer is to yes, the more value the translator adds
over what the user could already do with FreeCAD's STEP export.

We model our target on [bd_warehouse](https://github.com/gumyr/bd_warehouse)
— build123d's own author's curated component library — with two
deliberate departures (called out as `*` below).

---

## Imports

- **DO** import only the names actually used, one-per-line under
  black's auto-wrapping. `from build123d import Box, Cylinder, extrude`.
- **DON'T** `from build123d import *`. Idiomatic build123d code is
  specific even in consumer scripts; we should be specific too.
- **DO** put module helpers (`_edges_at`, `_pattern_union`, …) immediately
  below the imports and only emit them when used.
- *(eventual)* Helpers should be importable from `fcstd2b123d.runtime`
  rather than inlined; for now they're inlined to keep generated files
  self-contained.

---

## Top-of-file structure

```python
"""<one-line description from FreeCAD's Document.Label or Doc.Comment, when set;
otherwise: Auto-generated by fcstd2b123d from <relative path>.>"""

from build123d import <sorted, specific names>

# Optional: from build123d import Plane, Mode, ...  for builder-mode emits
# Optional: helper defs (when needed)

# --- geometry ---

<unit 1>

<unit 2>

result = <final var>
```

- One blank line between feature units.
- `result =` is contractual — tests and the REPL look for it.

---

## Names

- **DO** prefer the FreeCAD `Label` (the human name in the model tree)
  over `Name` (the autogenerated `Sketch001`) when they differ —
  snake-cased and de-shadowed.
- **DO** treat `Pad`, `Pocket`, etc. as feature kinds, not as variable
  names; the variable should describe *what it is*, not *how it was
  made*. `hex_head` > `pad_002`.
- **DON'T** let a translated variable shadow a build123d import. The
  snake_case post-pass handles this (`chamfer = chamfer(...)` →
  `chamfer_0 = chamfer(...)`); when adding new feature types, add the
  function name to `_B123D_FUNCTION_IMPORTS` in `emitter.py`.

---

## Patterns (Polar / Linear / Mirror)

Pattern features are the highest-impact area for "doesn't look auto-gen".

- **DO** detect uniform `Rot(Z=k·θ)` series and emit
  `with PolarLocations(radius, count): …` when there are ≥ 3 copies.
- **DO** detect uniform `Pos(i·dx, 0, 0)` series and emit
  `with GridLocations(dx, dy, nx, ny): …`.
- **DO** keep the `_pattern_union` / `_pattern_difference` helpers as
  the *fallback* when the locations don't fit a clean repeater (e.g.
  multi-Original patterns, irregular offsets, post-Mirror PolarPattern).
- **DON'T** spell out 50 `Rot(Z=k) * …` arguments when 5 lines of
  `with PolarLocations(): …` would do.

## Sketches and planes

Two styles, both acceptable:

### Algebra mode (default)

```python
profile = Polyline((0, 0), (10, 0), (10, 5), (0, 5), (0, 0))
sketch_a = make_face(profile)                            # XY at origin
sketch_b = Sketch() + Plane(origin=(0, 0, 5), x_dir=(1, 0, 0), z_dir=(0, 0, 1)) * (
    make_face(profile)
)
```

- The `Sketch() + …` wrap is for mypy resolution and stays even though
  it's line noise. Don't refactor it away unless we move the whole
  module to builder mode.

### Builder mode (`--style=builder`, future opt-in)

```python
with BuildSketch(Plane.XY.offset(5)) as sketch_b:
    Polyline((0, 0), (10, 0), (10, 5), (0, 5), (0, 0))
    make_face()
```

- Use `with BuildSketch(plane):` when the sketch sits on a non-XY plane
  — it's clearer than `Plane(origin=..., x_dir=..., z_dir=...)`.
- `make_face(mode=Mode.SUBTRACT)` instead of `(outer) - make_face(inner)`
  for holes inside a sketch.

## Literals and units

- **DO** emit dimensions snapped to FP-roundoff scale (`-19.999999999993534`
  → `-20`) but **DON'T** snap real solver-computed values (#43).
- **DO** preserve the FreeCAD-side digit count for arc parameters and
  centre coordinates that the constraint solver chose. The naive snap
  breaks BRep validity (documented in `format_value`).
- *(eventual)* Multiply dimensional literals by an emitted
  `MM = 1.0` / `IN = 25.4` constant for unit-readability. Gated on a
  CLI flag because it changes test snapshot bytes.

---

## Comments

- **DO** preface every feature unit with a one-line provenance comment
  carrying the FreeCAD TypeId, Label, and salient properties:
  `# PartDesign::Pad 'Hex head': length=4.7752`.
- **DO** prefer the FreeCAD Label (`'Hex head'`) over the Name (`'Pad'`)
  in the comment — it's what the modeller chose to call it.
- **DON'T** comment what the code does (`# Extrude the sketch`) — the
  build123d call is self-explanatory.
- **DON'T** leave TODOs or "this is approximate" notes without a reason;
  if the emit is approximate, log it as an issue and reference the
  issue number in the comment.

---

## Module-level shape

- **DO** keep emit at module level (flat statements) so the file is
  executable as `python part.py` and the result lands in a `result`
  variable. This is the translate-and-run contract.
- **DO** wrap in a `def make_part(*, …):` function *only* when the
  source FreeCAD doc has Spreadsheet parameters — then the parameters
  become keyword args with the source values as defaults.
- **DON'T** subclass `BasePartObject`. bd_warehouse does, but a
  bd_warehouse part is a *library component* meant to be instantiated;
  ours is a *translated drawing* meant to be modified. Different needs.

---

## Helpers

- **DO** factor any non-trivial routine that recurs across translators
  into `HELPER_DEFINITIONS` in `emitter.py`. Helpers are emitted at
  most once per output file and only when their key is added to the
  `TranslationUnit.helpers` set.
- **DO** put a real docstring on each helper explaining *why* it exists
  — the helpers are the bit of our output most likely to confuse a
  human reader who hasn't read this guide.
- **DON'T** import inside helpers unless lazy import is genuinely
  needed; black collapses an `import` inside the helper body into the
  module-level import block during formatting anyway.

---

## What we deliberately don't do

These are conscious departures from bd_warehouse style:

1. **No classes per Body.** We emit imperative module-level code. A
   bd_warehouse-style class wrapper would mean every output file must
   `import build123d` just to *import* (not just to run), which breaks
   the "open in a viewer and tweak" use case.
2. **No author / license / version metadata.** Translated files inherit
   the FreeCAD source's metadata via the docstring; padding with a
   pseudo-author block would be a lie.
3. **No CSV-driven parameter tables.** Tier-6 spreadsheet
   parametrisation is emitted as `def make_part(**defaults): …`
   function signatures, which keeps each translation self-contained.

---

## Self-check before committing a translator change

Generate a sample emit (`python -m fcstd2b123d <fixture>` on a
representative file) and ask:

1. Could I reasonably *delete* the auto-generated header and pass this
   off as hand-written? If no, what stops the illusion?
2. Are the variable names *about* the part, or *about FreeCAD*?
3. Does any pattern feature spell out copies instead of using `Locations`?
4. Are there helper definitions that should have been imported?
5. Does the `result =` line still exist?

If a translator change makes any of those answers worse, push back
before merging.
